= Automatische Volumenerweiterung
:allow-uri-read: 


Die automatische Volumenerweiterung ermöglicht es, dass von Trident bereitgestellte Persistent Volumes (PVs) automatisch wachsen, wenn ihre genutzte Kapazität einen definierten Schwellenwert erreicht.

Diese Funktion wird mithilfe von *Autogrow Policies (AGPs)* implementiert. Eine Autogrow Policy definiert:

* Die Auslastungsschwelle, die die Erweiterung auslöst
* Die Menge, um die das Volumen wächst
* Die maximale Größe, die das Volume erreichen kann


Die automatische Volumenerweiterung reduziert den Betriebsaufwand und hilft, Anwendungsunterbrechungen zu verhindern, die durch volle Volumes verursacht werden.



== Anforderungen

* Trident 26.02 oder später
* RBAC-Berechtigungen zum Erstellen `TridentAutogrowPolicy`benutzerdefinierter Ressourcen
* StorageClasses muss beinhalten `allowVolumeExpansion: true`
* Unterstützte ONTAP-Protokolle:
+
** NFS
** ISCSI
** FCP
** NVMe





NOTE: ONTAP NVMe Raw Block Volumes vor ONTAP 9.16.1 unterstützen keine automatische Erweiterung.


NOTE: Für SAN-Volumes, wenn  `growthAmount` kleiner oder gleich 50 MiB ist, erhöht Trident den Wert auf 51 MiB, bevor die Größe angepasst wird, vorausgesetzt, die resultierende Größe überschreitet nicht  `maxSize`.


NOTE: In Brownfield-Umgebungen funktioniert die automatische Erweiterung möglicherweise nicht für bestimmte vorhandene Volumes aufgrund des Migrationsverhaltens bei der Volume-Veröffentlichung.



== Erstellen Sie eine Autogrow-Richtlinie

Autogrow Policies sind Kubernetes-Benutzerressourcen vom Typ `TridentAutogrowPolicy`.

Eine Autogrow Policy definiert, wann und wie sich Volumes erweitern.



=== Felddefinitionen

[cols="1,3,1"]
|===
| Feld | Beschreibung | Erforderlich 


| `metadata.name` | Eindeutige Richtlinienkennung. | Ja. 


| `usedThreshold` | Prozentsatz der genutzten Kapazität, der eine Erweiterung auslöst. | Ja. 


| `growthAmount` | Betrag, um den beim Erreichen des Schwellenwerts erhöht wird. Kann als Prozentsatz (zum Beispiel, `"10%"`) oder als fester Wert (zum Beispiel, `"5Gi"`) angegeben werden. Standardmäßig  `10%` wenn nicht angegeben. | Nein 


| `maxSize` | Maximale Volumengrößenbegrenzung. Wenn nicht angegeben, kann das Volumen ohne eine festgelegte Obergrenze wachsen. | Nein 
|===


=== Beispiel: Standard Autogrow Policy

Das folgende Beispiel erstellt eine Richtlinie, die bei einer Auslastung von 80% eine Erweiterung auslöst, um 10% wächst und das Wachstum auf 500Gi begrenzt.

[source, yaml]
----
apiVersion: trident.netapp.io/v1
kind: TridentAutogrowPolicy
metadata:
  name: standard-autogrow
spec:
  usedThreshold: "80%"
  growthAmount: "10%"
  maxSize: "500Gi"
----


=== Erstellen Sie die Richtlinie

. Speichern Sie die YAML-Definition in einer Datei.
. Die Richtlinie anwenden:


[listing]
----
kubectl apply -f autogrow-policy.yaml
----
. Erstellung überprüfen:


[listing]
----
kubectl get tridentautogrowpolicy standard-autogrow
----
Erwartete Ausgabe:

[listing]
----
NAME                USED THRESHOLD   GROWTH AMOUNT   STATE
standard-autogrow   80%              10%             Success
----


== Konfigurationsbeispiele



=== Konservative Richtlinie für Produktionsdatenbanken

Verwenden Sie einen niedrigeren Schwellenwert und ein größeres Wachstumsinkrement für Produktionsdatenbank-Workloads, bei denen die Vermeidung von Kapazitätserschöpfung kritisch ist.

[source, yaml]
----
apiVersion: trident.netapp.io/v1
kind: TridentAutogrowPolicy
metadata:
  name: production-db-policy
spec:
  usedThreshold: "75%"
  growthAmount: "20%"
  maxSize: "5Ti"
----


=== Protokollspeicherung mit festem Wachstum

Verwenden Sie eine feste Schrittgröße, wenn vorhersehbare Wachstumsschritte bevorzugt werden.

[source, yaml]
----
apiVersion: trident.netapp.io/v1
kind: TridentAutogrowPolicy
metadata:
  name: log-storage-policy
spec:
  usedThreshold: "90%"
  growthAmount: "10Gi"
  maxSize: "100Gi"
----


=== Automatische Wachstumsrichtlinie mit Standardeinstellungen

Wenn nur  `usedThreshold` angegeben ist,  `growthAmount` ist der Standardwert  `10%` und  `maxSize` ist unbegrenzt.

[source, yaml]
----
apiVersion: trident.netapp.io/v1
kind: TridentAutogrowPolicy
metadata:
  name: logs-policy
spec:
  usedThreshold: "85%"
----


=== Autogrow-Richtlinie mit Standard-growthAmount

Geben Sie eine maximale Größe an, während Sie das Standardwachstumsinkrement verwenden.

[source, yaml]
----
apiVersion: trident.netapp.io/v1
kind: TridentAutogrowPolicy
metadata:
  name: default-ga-policy
spec:
  usedThreshold: "70%"
  maxSize: "100Gi"
----


=== Autogrow-Richtlinie mit Standard-maxSize

Ermöglichen Sie uneingeschränktes Wachstum, während Sie einen benutzerdefinierten Wachstumsprozentsatz festlegen.

[source, yaml]
----
apiVersion: trident.netapp.io/v1
kind: TridentAutogrowPolicy
metadata:
  name: default-ms-policy
spec:
  usedThreshold: "80%"
  growthAmount: "150%"
----


=== Autogrow-Richtlinie mit Bruchteilprozenten

Bruchteile von Prozentwerten ermöglichen eine granulare Steuerung von Schwellenwerten und Wachstumsschritten.

[source, yaml]
----
apiVersion: trident.netapp.io/v1
kind: TridentAutogrowPolicy
metadata:
  name: fractional-policy
spec:
  usedThreshold: "80.28%"
  growthAmount: "10.65%"
  maxSize: "100Gi"
----


== Richtlinienstatus

[cols="1,3"]
|===
| Status | Beschreibung 


| Erfolg | Richtlinie validiert und einsatzbereit. 


| Fehlgeschlagen | Validierungsfehler festgestellt. Überprüfen und korrigieren Sie die Spezifikation. 


| Löschen | Löschvorgang läuft. Finalisierer verhindern die Entfernung, solange Volumes auf die Richtlinie verweisen. 
|===


== Autogrow-Richtlinien verwalten



=== Richtlinien auflisten

[listing]
----
kubectl get tridentautogrowpolicy
----


=== Richtliniendetails anzeigen

[listing]
----
kubectl describe tridentautogrowpolicy production-db-policy
----


=== Verwendung von tridentctl

[listing]
----
tridentctl get autogrowpolicy
tridentctl get autogrowpolicy production-db-policy -o yaml
----


=== Eine Autogrow-Richtlinie aktualisieren


IMPORTANT: Änderungen betreffen alle Volumes, die derzeit die Richtlinie verwenden.

[listing]
----
kubectl edit tridentautogrowpolicy production-db-policy
----
Die Aktualisierungen werden beim nächsten Wachstumsvorgang angewendet. Ein Neustart des Volumes ist nicht erforderlich.



=== Löschen einer Autogrow-Richtlinie

Autogrow-Richtlinien beinhalten Finalizer-Schutz, um die Entfernung während der Nutzung zu verhindern.

[listing]
----
kubectl delete tridentautogrowpolicy production-db-policy
----
Wenn Volumes auf die Richtlinie verweisen, bleibt sie im  `Deleting` Zustand, bis alle Verweise entfernt wurden.

Um Volumes mithilfe einer Richtlinie zu identifizieren:

[listing]
----
tridentctl get autogrowpolicy production-db-policy -o yaml
----
Richtlinie von Volumes entfernen:

[listing]
----
kubectl annotate pvc <pvc-name> \
  trident.netapp.io/autogrowPolicy="none" \
  --overwrite
----


== Konfigurieren der Autogrow-Richtlinie auf einer StorageClass

Ordnen Sie eine Richtlinie mithilfe der  `trident.netapp.io/autogrowPolicy` Annotation zu.

[source, yaml]
----
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: ontap-gold
  annotations:
    trident.netapp.io/autogrowPolicy: "production-db-policy"
provisioner: csi.trident.netapp.io
parameters:
  backendType: "ontap-san"
  fsType: "ext4"
allowVolumeExpansion: true
----
Annotation überprüfen:

[listing]
----
kubectl get storageclass ontap-gold \
  -o jsonpath='{.metadata.annotations.trident\.netapp\.io/autogrowPolicy}'
----


=== Autogrow-Richtlinie für eine StorageClass aktualisieren

[listing]
----
kubectl annotate storageclass ontap-gold \
  trident.netapp.io/autogrowPolicy="standard-autogrow" \
  --overwrite
----


=== Entfernen Sie die Autogrow-Richtlinie aus einer StorageClass

Option 1: Annotation entfernen

[listing]
----
kubectl annotate storageclass ontap-gold \
  trident.netapp.io/autogrowPolicy-
----
Option 2: Annotation auf  `none` setzen

[listing]
----
kubectl annotate storageclass ontap-gold \
  trident.netapp.io/autogrowPolicy="none" \
  --overwrite
----


== Wenden Sie die Autogrow-Richtlinie auf ein PVC an

Wenden Sie eine Richtlinie auf PVC-Ebene an, um das Verhalten von StorageClass zu überschreiben.

[source, yaml]
----
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: database-pvc
  annotations:
    trident.netapp.io/autogrowPolicy: "production-db-policy"
spec:
  storageClassName: ontap-gold
  accessModes:
    - ReadWriteOnce
  resources:
    requests:
      storage: 500Gi
----
Autogrow auf einem PVC deaktivieren:

[listing]
----
kubectl annotate pvc database-pvc \
  trident.netapp.io/autogrowPolicy="none" \
  --overwrite
----


== Autogrow-Richtliniennutzung überwachen

Richtlinien auflisten:

[listing]
----
kubectl get tridentautogrowpolicy
----
Finden Sie heraus, welche Richtlinie ein PVC verwendet:

[listing]
----
kubectl get pvc database-pvc \
  -o jsonpath='{.metadata.annotations.trident\.netapp\.io/autogrowPolicy}'
----
Ereignisse überwachen:

[listing]
----
kubectl get events \
  --field-selector involvedObject.kind=TridentAutogrowPolicy
----


== Fehlerbehebung

[cols="1,3,2"]
|===
| Problem | Mögliche Ursache | Auflösung 


| Policy im  `Failed` Staat | Ungültige Spezifikation | Korrigieren Sie die Richtlinie und wenden Sie sie erneut an 


| Volume wird nicht erweitert | `maxSize` erreicht | Erhöhen `maxSize` 


| Häufige Erweiterungen | Schwellenwert zu niedrig | Erhöhen `usedThreshold` 


| PVC zeigt `AutogrowPolicyNotFound` | Richtlinie existiert nicht | Richtliniennamen überprüfen 


| PVC zeigt `AutogrowPolicyNotUsable` | Policy im  `Failed` Staat | Beheben Sie Validierungsfehler 


| TAGRI wiederholt abgelehnt | `maxSize` überschritten | Erhöhen `maxSize` oder anpassen `growthAmount` 
|===
